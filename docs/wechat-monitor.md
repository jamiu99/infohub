# 公众号监控：产品形态 · UI/UX · 调度

> 上级：[overview.md](overview.md) · 登录/账号/限流底层见 [wechat-login.md](wechat-login.md) · 采集接口见 [ingest.md](ingest.md#微信公众号)
> 这是 P1 的**主线功能与产品核心**：让用户监控一批公众号，自动发现新文章。

## 一、产品形态

核心是一个**监控循环**：

```
维护关注列表（一批公众号）
   → 定时轮询每个号的 appmsg 接口发现新文章（也可手动刷新单个号）
   → 新文章入库、标未读，用户在文章流里看
```

规模假设：**几十个公众号**。这个量级下单账号每小时配额（~50 请求）不够，**多账号轮换 + 排队调度是刚需**（见[四、调度]）。

## 二、主界面：三栏布局

```
╭──────────────┬─────────────────────┬──────────────────────╮
│ 源列表        │ 文章流               │ 文章详情              │
│              │ (选中号或"全部")      │                       │
│ ● 特工宇宙 ③  │ ┌─────────────────┐ │  标题                 │
│ ○ 量子位      │ │● 新文章 A   2h前 │ │  作者·来源·发布时间    │
│ ● 机器之心 ①  │ │  文章 B     昨天 │ │  ─────────────       │
│ ⚠ 某号(失效)  │ │  文章 C     3天前│ │  正文 markdown        │
│ ──────────   │ └─────────────────┘ │                       │
│ + 加公众号    │                     │  [原文] [摘要] [归档]  │
├──────────────┤                     │                       │
│ 账号池 & 配额  │                     │                       │
│ ▓▓▓░ 32/50   │                     │                       │
╰──────────────┴─────────────────────┴──────────────────────╯
```

### 左栏：源列表 + 账号池状态

- 每个公众号一行：**状态圆点 + 名称 + 未读数徽标**。
  - `●` 有新未读 / `○` 无未读 / `⚠` 该号最近抓取失败（如所有账号限流/登录失效）
- 顶部"全部"聚合视图；`+ 加公众号`走搜索添加流程（见[三]）。
- **左栏底部常驻账号池 & 配额条**（监控场景的关键信息，必须一眼可见）——见[五、UX 重点]。

### 中栏：文章流

- 选中某号 → 显示该号文章；选"全部" → 所有号按时间倒序聚合。
- 每条：**未读点 + 标题 + 相对时间**（"2h 前"，存 UTC 展示本地时区）+ 可选封面缩略。
- 未读加粗；点击即读，滚动已读可选。
- 顶部：手动刷新按钮（刷当前号）、筛选（未读/全部/已归档）。

### 右栏：详情

- 标题 / 作者 / 来源号 / 发布时间 / 原文链接。
- 正文 markdown 渲染（正文抓取见 [process.md](process.md)）。
- 操作：打开原文、生成摘要（走 [agent](agent.md)）、归档、打标签。

## 三、添加公众号流程

```
点 "+ 加公众号" → 输入名称 → 调 searchbiz 搜索（用账号池里一个 active 账号）
  → 展示候选（头像/nickname/alias/signature）→ 用户选中
  → 存为 Source{ type:wechat, config:{fakeid}, name }
  → 立即触发一次增量抓取，填充文章流
```

搜索也吃配额（每次 searchbiz 算一次请求），走同一调度器。

## 四、轮询与调度

**触发方式**：定时自动 + 手动刷新（用户选定）。

- **定时自动**：后台按可配间隔（如每 2-4 小时）轮询全部关注号的增量（每号 `maxPages≈3`）。间隔受配额约束——几十个号 × 3 页远超单账号小时配额，所以一轮会被调度器摊到多个账号 + 跨时间排队。
- **手动刷新**：用户对单个号或"全部"点刷新，插队优先执行（仍受限流保护）。

### 调度器职责（core，见 [wechat-login.md](wechat-login.md#四多账号与限流)）

```
轮询任务队列（每号一个增量任务）
   → 调度器按账号池挑 active 且未超小时配额的账号执行
   → 请求间隔 sleep、账号间轮换、计数
   → 命中 200013 → 账号 cooldown 2h，任务回队列换号
   → cookie 失效 → 账号 expired，触发扫码引导；该号标 ⚠
   → 全账号不可用 → 任务排队，UI 显示"等待配额恢复至 HH:MM"
```

**关键**：轮询不是"一次跑完"，而是**持续消耗配额的长队列**。UI 要如实反映"进度 + 预计完成时间 + 配额剩余"，不能假装瞬间刷新。

## 五、UX 重点（监控场景的成败在这两处）

### 1. 配额可视化 —— 让用户看懂"还能爬多少"

左栏底部常驻，点开是账号池面板：

```
账号池 (2 个)                         下一轮: 14:20
┌────────────────────────────────────────────┐
│ 账号A  ▓▓▓▓▓▓▓░░░  32/50  active           │
│ 账号B  ▓▓░░░░░░░░   8/50  active           │
│ 账号C  ⏸ cooldown 至 15:30 (命中限流)       │
│ 账号D  ⚠ 登录失效  [重新扫码]               │
│ + 添加账号（提升每日采集上限）               │
└────────────────────────────────────────────┘
```

- 每账号一条配额进度条（本小时已用/上限）+ 状态。
- **配额是硬约束，必须显性**：让用户理解"加账号 = 提升采集上限"，自然引导多账号。
- cooldown 显示恢复时刻；失效显示 `[重新扫码]` 按钮。

### 2. 登录失效引导 —— 别让用户一脸懵

- 账号失效时：该号在左栏标 `⚠`，账号池对应条目变红 + `[重新扫码]`。
- 点击 → 打开该账号 `persist:` 分区的 BrowserWindow（见 [wechat-login.md](wechat-login.md#五cookie-失效检测与扫码引导)），很多时候页面还在只需刷新确认。
- 全局提示（非阻塞 toast）："账号 X 登录失效，采集已暂停该账号，点此重新扫码。"
- **主动保活**：每天低频探测提前发现失效，避免轮询时才暴雷。

### 3. 首次空状态

无任何公众号时，中栏引导：先登录一个账号（扫码）→ 再搜索添加第一个公众号 → 自动抓取展示。三步走完就有内容，建立正反馈。

## 六、数据落点

- 关注的公众号 = `Source{type:wechat}`，存 `data/sources.json`（见 [storage.md](storage.md)）。
- 文章经 [process](process.md) 归一为 `Article`，落 `data/articles/wechat/<fakeid>/`。
- 未读/已读/归档状态存 SQLite 索引（`articles` 表加 `read`/`archived` 字段——实现时补 schema，同步更新 [storage.md](storage.md)）。
